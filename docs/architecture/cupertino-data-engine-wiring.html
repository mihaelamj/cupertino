Cupertino Data Engine Wiring

Current state: CupertinoDataEngine is an external package. CupertinoComposition is still inside the cupertino repo and adapts the current production readers into that external facade. Desktop and mobile UI code should only see backend contracts or the engine facade, never storage paths or concrete reader types. Mobile clients may download or bundle an opaque Cupertino corpus, but Cupertino-owned engine code owns how that corpus is opened.

Target direction: move source, sample, and package read-storage into CupertinoDataEngine so CupertinoComposition becomes thinner. Mobile apps can download or bundle a Cupertino corpus, but only the Cupertino-owned engine opens the resources inside it; app clients still never know that storage exists.
