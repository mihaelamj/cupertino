Cupertino Data Engine Wiring

Current state: CupertinoDataEngine is an external package. CupertinoComposition is still inside the cupertino repo and supplies schema-version policy, but app-facing construction now passes an opaque CupertinoDataEngine.Corpus handle. The engine owns resource filenames plus read-only source, sample, and package readers. Desktop and mobile UI code should only see backend contracts or the engine facade, never storage paths or concrete reader types.

Next direction: prove the complete engine closure on an iOS destination and wire desktop/mobile catalog state to provide opaque corpus handles. Mobile apps can download or bundle a Cupertino corpus, but only the Cupertino-owned engine opens the resources inside it; app clients still never know that storage exists.
