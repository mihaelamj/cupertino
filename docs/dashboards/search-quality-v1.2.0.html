Cupertino · v1.2.0 candidate

Does the search find the right answer?

Cupertino is an Apple-platform documentation index built to keep AI coding assistants from hallucinating. This page summarises seven tests of how well it does that on the v1.2.0 build.

352,712 documents·420 frameworks·Measured 2026-05-20

What we found

Cupertino is strong at the queries it was tuned for — typing a Swift type name returns the canonical Apple documentation page at position 1 nine times out of ten, and the modern Swift form always wins over the legacy Objective-C form. Two tests surface real weaknesses: abbreviations (typing NFC instead of CoreNFC) and attribute-only queries (typing initializer or subscript) often fail to return the right pages. Both weaknesses point at the same architectural gap — the same fix can address both.

Strong

Looking up a Swift type by name

Test A + B · Canonical lookup & framework root

92%

right answer at position 1

When you type Hashable, URLSession, or LazyVGrid, does the canonical Apple documentation page come up first?

Out of 50 typed type-names, the canonical Apple page came up first for 46. One didn't return the right page at all. The three other near-misses had the right page at position 2 or 3.

Strong

Modern Swift over legacy Objective-C

Test E · Deprecation awareness

100%

modern form wins every time

When you type URL, does the modern URL value type come up above the legacy NSURL class?

All 30 pairs tested. The modern Swift form was at position 1 for every single one. The legacy NS-prefixed form was either absent from the top 10 or buried at position 2-8. This is the single most user-visible anti-hallucination axis.

Strong (with caveat)

Choosing the right source

Test F · Cross-source canonical answer

19/19

when the right source has any page at all

When multiple Apple sources (docs, Swift Evolution, HIG) all have something to say, does cupertino pick the most authoritative one?

For all 19 queries where the expected source was even in the top 10, that source's hit was at position 1. The remaining 6 queries asked things only HIG or Apple Archive answers well — those got drowned out by Apple-docs because the system is intentionally tuned to favour API references over design guidance.

Weak

Abbreviated framework names

Test C · Acronym & synonym routing

18%

canonical framework at position 1

When you type NFC, wifi, bluetooth, or ml, does cupertino take you to CoreNFC, CoreWLAN, CoreBluetooth, or CoreML?

Only 4 of 22 abbreviations routed to their canonical framework at position 1. The synonym table exists in the database but the ranking layer doesn't appear to consult it for free-text queries. NFC lands on a deep authentication-services page that happens to contain the word "nfc", not the CoreNFC framework.

Weak

Finding code by what it is

Test H · Symbol attributes & kinds

25%

average match rate in top 5

When you ask for @MainActor properties, initializers, or subscripts, does cupertino return pages that actually have those things?

Mixed results. Queries where the attribute is also a Swift type name (@MainActor, Sendable conformance) work well. Queries that only describe a symbol kind (initializer, subscript, typealias) return zero relevant results. Same architectural gap as the acronym test: the relational symbol metadata isn't reached by the default search path.

Mixed

Natural-language questions

Test G · Prose & conceptual queries

27%

strict; ~55-65% useful-but-loose

When you type a full question like how does Observable invalidate views or actor reentrancy semantics, do useful results come up in the top 3?

By the strict measure, 4 of 15 prose queries returned exactly the document we expected. Reading the misses honestly, about half of them returned tangentially-useful pages a human would accept. The system is tuned for short keyword queries; longer multi-word questions need a different ranking profile.

Coming next

Does an AI agent then write correct Swift?

Test 1.7 · Anti-hallucination, end-to-end

—

design exists, implementation pending

All seven tests above measure whether the right document is findable. This test measures whether an AI coding agent, given cupertino's top results, actually produces Swift that compiles, calls real APIs, and respects platform availability.

This is the real success measure for cupertino. The design is written; the implementation is the next 1-2 weeks of work. Until it lands, the anti-hallucination claim is unmeasured end-to-end.

Read the design

One architectural pattern surfaces twice

The two weak tests (C abbreviations and H symbol attributes) both reveal the same gap: cupertino has rich relational metadata (synonym tables, symbol kinds, attributes, conformances) that the default search path doesn't consult. A consolidated fix for one likely fixes both. Tracking issues are open with three candidate approaches each, ranging from a small CLI flag to a full schema augmentation.
