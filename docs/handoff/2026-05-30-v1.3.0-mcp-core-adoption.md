## What is being released, and what it entails

### 1. `swift-mcp-core` adoption (#1167) — the headline

cupertino's MCP wire-protocol types (the `MCP.Core.Protocols.*` family: JSON-RPC envelopes, `Tool`, `ContentBlock`, `Implementation`, the Prompts family, capabilities, …) have been extracted into a **new neutral public Swift package**, `swift-mcp-core` (module **`SwiftMCPCore`**, tag **v0.1.0**), shared with the sibling `CupertinoMCPClientKit` + `cupertino-desktop` repos.

**What changes inside cupertino (the carve-out):**
- The `MCPCore` target **deletes** its 8 in-repo wire-type files + the `MCP` / `MCP.Core` / `MCP.Core.Protocols` namespace anchor.
- `MCP.swift` becomes `@_exported import SwiftMCPCore` (re-exporting the wire types under the **identical** `MCP.Core.Protocols.*` namespace, so the ~6 consumer targets need **zero** call-site edits) plus `extension MCP.Core { enum Transport {} }` (the package owns `Protocols` but not `Transport`).
- cupertino **keeps** its Server + Transport layers (they extend the package's anchor). `CupertinoIcon` stays cupertino-local.
- `Packages/Package.swift` gains `.package(url: "https://github.com/mihaelamj/swift-mcp-core.git", from: "0.1.0")` and `MCPCore` depends on the `SwiftMCPCore` product.

**What it ENTAILS (release implications — read before tagging):**
- cupertino gains its **first non-Apple external dependency**. CI and every from-source build now resolve `swift-mcp-core` from GitHub. The prebuilt binary (Homebrew / `install.sh`) is unaffected for end users; this touches CI / source builds / the release pipeline.
- **Version coupling:** cupertino releases pin a `swift-mcp-core` tag (currently `0.1.0`). A wire-protocol change means a `swift-mcp-core` bump + a cupertino dep bump. Desktop agreed to treat `swift-mcp-core` as a published-API package (no breaking wire changes without a version bump + a heads-up on the coord channel).
- **De-risked, not theoretical:** the spike branch performed the full carve-out and proved it green — `swift build` + 67 unit tests (`MCPTests` 43 + `MCPSupportTests` 24) + a **real end-to-end `cupertino serve` session** (initialize + tools/list of 12 tools + a real `tools/call` returning live results), all against the published `swift-mcp-core@0.1.0`.

**Decision history:** first planned for v1.4.0 (held to keep v1.3.0 lean), then reversed to **adopt now in v1.3.0** once `v0.1.0` was tagged + proven. v2.0 was briefly floated, then walked back to v1.3.0.

### 2. `serve` per-source search fix (#1168)

Post-#1036 (the per-source DB split) the monolithic `search.db` is no longer built, but `cupertino serve` still resolved its primary search index to `paths.searchDatabase` (`<baseDir>/search.db`). On a per-source bundle (which can carry an empty `search.db` stub), serve opened the stub, so `search` / `list_frameworks` returned **0 results** despite fully-populated per-source DBs.

**Fix:** serve resolves the apple-docs primary index through the production source registry (the apple-docs provider's `destinationDB.filename`) — not a hardcoded filename and not the legacy `search.db` (Source Independence axiom; falls back to the legacy path only if the apple-docs provider is absent). It **also adds `--base-dir`** so a host can aim the MCP server at any bundle without a `cupertino.config.json`. Verified E2E: `serve --base-dir <bundle>` returns real results (`NavigationStack` → `apple-docs://swiftui/navigationstack`; `list_frameworks` → 354,963 docs). #1071 family.
